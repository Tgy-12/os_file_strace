Robust File Copying & Kernel Verification (Ubuntu)

This interactive document synthesizes the design rationale and implementation of a file copy program using the low-level POSIX API, and verifies its system interaction using the strace utility on Ubuntu, fulfilling the requirements for Lab 1 and Lab 3.

01 Design Decisions

The core decision for this project was maximizing efficiency by minimizing the number of expensive context switches between user space and kernel space. This led to two key choices: using the direct POSIX system calls (open, read, write, close) and optimizing the buffer size.

Buffer Size Optimization: Why 4KB?

The 4096-byte (4KB) size is chosen because it precisely matches the standard Virtual Memory Page Size on x86/x64 Linux systems, which is the unit of data the kernel prefers to handle, leading to the lowest relative overhead.

API Selection: POSIX over StdLib

Using functions like fread() and fwrite() introduces an unnecessary layer of buffering in user space. The POSIX system calls (like read) bypass this extra layer, allowing us to manage the buffer size directly and minimize context switches, which are expensive CPU operations.

File Descriptors (FDs)

Unlike the high-level FILE* streams, POSIX functions operate on integers known as File Descriptors. FD 0, 1, and 2 are pre-assigned to standard input, output, and error, respectively. Our program uses the next available FDs (typically 3 and 4) for the source and destination files.

02 Execution & Error Flow

The program logic is structured into distinct phases, with explicit checks for critical errors at each point, as required for robustness. Click on any step to view the specific C code snippet and the type of error handling implemented.

1. Open Source File (open)

O_RDONLY

Attempt to acquire the source file descriptor for reading only.

int src_fd = open(source_path, O_RDONLY);
if (src_fd == -1) {
  perror("Source file error (e.g., Not Found, Permission Denied)");
  exit(1);
}
Handles: ENOENT, EACCES

2. Create/Truncate Destination (open)

O_WRONLY|O_CREAT|O_TRUNC

Opens the destination, creating it with 0644 permissions if missing, or truncating existing content.

int dest_fd = open(dest_path, O_WRONLY|O_CREAT|O_TRUNC, 0644);
if (dest_fd == -1) { close(src_fd); exit(1); }
Robustness: Source FD is closed before exit to prevent resource leaks.

3. The Main Copy Loop (read / write)

Loop Until EOF

The loop transfers data in 4KB chunks. Critical error handling ensures that both successful read operations and complete write operations are verified.

char buffer[BUF_SIZE]; // 4096
ssize_t n, written;
while ((n = read(src_fd, buffer, BUF_SIZE)) > 0) {
  written = write(dest_fd, buffer, n);
  if (written != n) {
    perror("Destination write error (e.g., Disk Full)");
    ... cleanup and exit ...
  }
}
if (n == -1) { perror("Source read error"); ... }
Handles: EDQUOT, EIO, Read Errors (e.g., interrupted system call)

4. Cleanup & Terminate (close)

Resource Release

Files must be explicitly closed to flush buffers and release the file descriptors back to the OS.

close(src_fd);
close(dest_fd);
return 0;

Trade-off Analysis

N / 4096

Total Read/Write Calls (Approximation)

~ 2%

Context Switch Overhead (Relative to 1B buffer)

The total number of read and write system calls is directly proportional to the file size (N) divided by the buffer size (4096 bytes).

**If the buffer size was 1 byte (N/1):** The system would incur two context switches per byte, leading to a massive CPU overhead and dramatically slow copying.

**With 4096 bytes:** We trade slightly increased memory usage for a massive reduction in system call frequency, resulting in optimal throughput for sequential I/O operations. The strace output confirms each call attempts to transfer a full 4096 bytes, as designed.
